# Парсер ecotechstroy.ru: анализ, архитектура и план реализации

## 1) Анализ сайта

### Ключевые наблюдения
- Сайт построен на Next.js (видны `_next/static/...`, RSC-запросы с параметром `_rsc`).
- Каталог доступен по адресу: `https://ecotechstroy.ru/catalog`.
- Карточки домов открываются по шаблону URL: `https://ecotechstroy.ru/catalog/<category>/<uuid>`.
- Для выдачи списка проектов сайт использует backend-эндпоинт:
  - `GET https://ecotechstroy.ru/apiback/cards`
- В `apiback/cards` уже присутствуют почти все требуемые поля:
  - `title` (название)
  - `area` (площадь)
  - `price` (цена)
  - `floors` (этажность)
  - `id` (идентификатор карточки)
  - `options` (флаги типа материала/категории)

### Как получить ссылку на карточку дома
Эндпоинт `apiback/cards` не отдает полный URL карточки, но дает `id` + `options`.
Ссылку можно детерминированно собрать:

1. Вычислить category slug по `options`:
   - `isTimberHouse=true` -> `kleeny-brus`
   - `isFrameHouses=true` -> `karkasny`
   - `isRoundedLogs=true` -> `ocilindrovannoe-brevno`
   - `isHandCutHouses=true` -> `ruchnaya-rubka`
   - `hasBathhouse=true` -> `bani`
2. Построить URL:
   - `https://ecotechstroy.ru/catalog/<slug>/<id>`

### Как получить материал
Материал можно получить двумя способами:
1. Быстрый способ: нормализовать `options` в человекочитаемое значение.
2. Точный способ: открыть карточку и прочитать поле из блока характеристик (например, `клеёный брус`, `финские дома` и т.п.).

Рекомендуем гибрид:
- Базово использовать `options -> material`.
- Для повышения качества данных делать валидацию/обогащение с детальной карточки.

## 2) Архитектура приложения

## Цель
Собрать унифицированный датасет домов со следующими полями:
- `name`
- `area_m2`
- `material`
- `price_rub`
- `floors`
- `card_url`

## Предлагаемые компоненты

### 2.1 Source Client (источник данных)
- Отвечает за HTTP-запросы к `https://ecotechstroy.ru/apiback/cards`.
- Настраиваемые `timeout`, `retry`, `backoff`, `user-agent`.
- Логирование статусов и ошибок.

### 2.2 Mapper/Normalizer
- Преобразует сырой JSON в доменную модель `House`.
- Конвертирует `options` в:
  - `category_slug`
  - `material` (дефолтное значение)
- Собирает `card_url` по шаблону.
- Нормализует числовые поля (float/int), фильтрует невидимые карточки (`isVisible`).

### 2.3 Enricher (опционально)
- Дообогащает данные из detail-page (если нужны 100% точные формулировки материалов/доп. атрибутов).
- Работает ограниченно по rate-limit.

### 2.4 Storage/Exporter
- Экспорт в `CSV` и/или `JSON`.
- Версионирование выгрузок по timestamp.
- Возможность upsert в БД (PostgreSQL) на следующем этапе.

### 2.5 Orchestrator (CLI/job)
- Команда запуска: `parse_once`.
- Планировщик (cron/Airflow) для периодических запусков.
- Метрики: сколько карточек получено/записано/отбраковано.

## 3) План реализации

### Этап 1. MVP (1-2 дня)
1. Реализовать клиент к `apiback/cards`.
2. Описать модель `House` + валидацию полей.
3. Реализовать маппинг `options -> material/category_slug`.
4. Добавить сборку `card_url`.
5. Сделать экспорт в `houses.json` и `houses.csv`.
6. Добавить базовые unit-тесты на маппинг.

### Этап 2. Качество и устойчивость (1 день)
1. Добавить retry/backoff и обработку нестандартных ответов.
2. Добавить checks на пустые/нулевые цены и площадь.
3. Добавить дедупликацию по `id`.
4. Добавить структурированные логи.

### Этап 3. Обогащение из карточек (опционально, 1-2 дня)
1. Пройтись по `card_url` и вычитать материал/детали из HTML.
2. Сверить с материалом из `options`.
3. Выбирать приоритетный источник (detail > options).
4. Добавить feature-flag `--enrich-details`.

### Этап 4. Продовый контур (опционально)
1. Инкрементальная загрузка в PostgreSQL.
2. Docker-упаковка.
3. Cron/CI запуск и алерты при падении.

## Риски и меры
- Изменение внутреннего API (`/apiback/cards`):
  - fallback на парсинг HTML каталога.
- Изменение структуры `options`:
  - словарь соответствий + тесты на неизвестные комбинации.
- Блокировки/ограничения:
  - умеренная частота запросов, ретраи, ротация user-agent (при необходимости).
